# Задача 6 - развёртывание `my-app` в Kubernetes

## Требования

Есть образ с сервисом, который локально запускается так:

```bash
docker run -d -p 8080:80 --name my-app my-app-image
```

Нужно развернуть его в Kubernetes так, чтобы:

- сервис был доступен снаружи кластера
- приложение автоматически восстанавливалось при падении
- приложение можно было масштабировать

## Использованные манифесты

- [deployment.yaml](deployment.yaml) - развертывание приложения, несколько реплик, пробы, rolling update
- [service.yaml](service.yaml) - публикация сервиса наружу через `LoadBalancer`
- [hpa.yaml](hpa.yaml) - автоматическое масштабирование по CPU

## Какое решение выбрано

### Deployment

Используется `Deployment`, так как приложение в условии смахивает как stateless-сервис (так обычно называют не БДшки и другие чувствительные к перезапуску и переносу программы):

- контейнер запускается из готового образа
- про состояние/volume в условии ничего не сказано :)
- легко масштабировать и автоматически восстанавливать Pod'ы

В `Deployment` указано:

- `replicas: 2` - базовая отказоустойчивость
- `RollingUpdate` - обновление без одновременного отключения всех Pod
- `readinessProbe` и `livenessProbe` - базовый контроль готовности и живости приложения (без startupProbe, так как мы ничего не знает о приложении и не знаем, как его чекать)
- `resources` - requests/limits для HPA, чтобы он мог работать по CPU

### Service

Для внешнего доступа используется `Service` типа `LoadBalancer` (нам ничего не сказано об архитектуре, я нафантазировал себе обалчный кластер с возможностью LB)

На основе `docker run -p 8080:80` считаем, что в Kubernetes:

- приложение внутри контейнера слушает `80`
- наружу сервис публикуется на `8080`

Тогда как раз `Service` перенаправляет внешний `port: 8080` на `targetPort: 80`

### HPA

Дополнительно добавлен `HorizontalPodAutoscaler`:

- `minReplicas: 2`
- `maxReplicas: 5`
- масштабирование по CPU (`averageUtilization: 70`)

## Почему выбрано именно так

Этим решением мы закрываем все требования задачи:

- **доступно снаружи кластера** - через `Service LoadBalancer`
- **устойчиво к сбоям** - `Deployment` поддерживает нужное число реплик, а probes помогают Kubernetes понимать, когда Pod готов к трафику и когда контейнер нужно перезапустить
- **легко масштабируется** - можно менять `replicas` вручную или использовать `HPA`

## Важные допущения (повторяюсь)

- приложение stateless
- контейнер действительно слушает `80/tcp`
- образ `my-app-image` доступен из кластера
- в кластере есть поддержка `Service` типа `LoadBalancer`
- в кластере есть источник метрик для `HPA`

## Что можно было бы сделать иначе

### Если в кластере нет поддержки `LoadBalancer`

Тогда внешний доступ можно организовать через `NodePort` (то есть открыть порты на всех нодах)

### Если нужен доступ по доменному имени и маршрутизация HTTP/HTTPS

`Ingress`, но для этого нужен ingress-controller

## Ограничения

Манифесты не тестировались на реальном кластере, так как в рамках временного окна тестового задания не было возможности поднять облачный Kubernetes-кластер и подготовить и опубликовать тестовый образ некоего приложения, который бы менял существенно нагрузку на процессор

Поэтому решение является теоретическим и отражает дух ожидаемого способа развёртывания такого сервиса
